    协同仿真模块¶

    协同仿真（Co-Simulation）是 tricys 中一项高级且强大的功能，它允许 Modelica 模型与外部程序（如 Aspen Plus, MATLAB, 或自定义的 Python 脚本）进行交互，实现混合仿真。

    这对于以下场景尤其有用： - 某个子系统的模型在另一个专业软件中已经存在且非常成熟。 - 子系统的行为非常复杂，难以用 Modelica 语言直接描述，但可以通过外部计算得到。

    工作流程¶

    协同仿真的核心思想是“运行-处理-再运行” (Run-Process-Rerun)。

    graph LR
    A[1. 初步仿真] --> B{结果};
    B --> C[2. 外部处理器(Handler)];
    C --> D{新数据};
    D --> E[3. 注入新数据];
    E --> F[4. 最终完整仿真];

    style A fill:#A6C8FF
    style C fill:#FFDDA6
    style F fill:#A6FFC8
    1. 初步仿真 (Primary Run): tricys 首先运行一个不包含被替代子系统的初步仿真。
    2. 调用处理器 (Call Handlers): 初步仿真的结果被传递给一个或多个用户定义的“处理器”(Handler)。
    3. 外部计算: 处理器函数接收结果，并调用外部程序（如 Aspen）或执行内部算法，计算出新的数据。
    4. 数据注入 (Injection): 处理器返回一个数据字典。tricys 会动态地创建一个新的顶层模型，用 CombiTimeTable (数据表) 模块替换掉原来的子系统，并将处理器返回的数据作为这个数据表的内容。
    5. 最终仿真 (Final Run): tricys 运行这个被动态修改过的、包含注入数据的新模型，从而完成一次完整的协同仿真。

    配置文件示例¶

    通过在 config.json 中定义 co_simulation 字段来启用此功能。

    {
    "paths": { ... },
    "simulation": { ... },
    "simulation_parameters": { ... },
    "co_simulation": [
        {
            "submodel_name": "example_model.DIV",
            "instance_name": "div",
            "handler_module": "tricys.handlers.div_handler",
            "handler_function": "run_div_simulation",
            "params": {
                "description": "This is a dummy handler for DIV.",
                "dummy_value": 123.45
            }
        },
        {
            "submodel_name": "example_model.I_ISS",
            "instance_name": "i_iss",
            "handler_module": "tricys.handlers.i_iss_handler",
            "handler_function": "run_aspen_simulation",
            "params": {
                "bkp_path": "../../example_aspenbkp/T2-Threetowers4.bkp",
                "base": 20,
                "retime": 60,
                "time_step": 3
            }
        }
    ]
}

    配置项详解¶

    co_simulation¶

    • 描述: 这是一个列表，其中每个元素都是一个“处理器配置对象”，用于定义如何处理一个特定的子模型。

    处理器配置对象¶

    • submodel_name (字符串, 必填):

    • 描述: 要被外部处理器替代的子模型的完整 Modelica 类路径。例如 example_model.DIV。

    • instance_name (字符串, 必填):

    • 描述: 该子模型在顶层仿真模型 (simulation.model_name) 中的实例名。例如，如果顶层模型中有 DIV div;，那么这里的 instance_name 就是 div。

    • handler_module (字符串, 必填):

    • 描述: 包含处理器逻辑的 Python 模块的路径。

    • handler_function (字符串, 必填):

    • 描述: 要调用的处理器函数名。

    • params (字典, 选填):

    • 描述: 传递给处理器函数的额外参数。

    编写您自己的处理器¶

    您可以编写自定义的 Python 函数来作为处理器。

    处理器函数的规范: 1. 输入: - 函数的第一个参数必须是 df，用于接收初步仿真的结果（一个 Pandas DataFrame）。 - 函数的第二个参数必须是 sim_info，一个包含当前仿真信息的字典。 - 其余参数可以通过 JSON 文件中的 params 字段自由定义。

    1. 输出:
      • 函数必须返回一个字典。
      • 该字典的键是 Modelica 变量的完整路径（例如 div.ports[1].F）。
      • 该字典的值是一个包含两列的 numpy 数组或列表：第一列是时间，第二列是该时间点对应的变量值。这个数据将被用来生成 CombiTimeTable。

    示例函数签名: 

    def my_handler_function(df, sim_info, my_param1, my_param2):
    # 1. 从 df 中提取所需输入
    # 2. 执行外部计算或自定义逻辑
    # 3. 准备要返回的数据

    time_data = [0, 100, 200]
    value_data = [1.0, 1.5, 1.2]

    # 将时间和值合并成一个 (N, 2) 的数组
    output_table = np.vstack((time_data, value_data)).T

    return {
        "div.ports[1].F": output_table,
        "div.ports[1].T": output_table_for_T  # 另一个变量的数据
    }

    通过这种方式，您可以将任何外部计算无缝集成到 tricys 的仿真流程中，实现高度复杂的系统级分析。
